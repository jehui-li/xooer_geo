建议按以下方式提供提示词，便于高效推进：

## 推荐的提示词结构

### 1. 阶段化任务分解
每次只给一个阶段或一个模块的任务，例如：
- "开始第一阶段：搭建项目基础架构，创建项目结构、配置环境变量管理、设置日志系统"
- "实现第二阶段：开发探针指令集生成器，支持三类探针模板"
- "完成第三阶段：实现实体识别模块，使用 spaCy 提取品牌提及信息"

### 2. 明确任务边界
每次任务需包含：
- 具体要做什么（功能点）
- 不要做什么（避免范围蔓延）
- 输出要求（文件、接口、数据结构）

示例：
```
任务：实现 OpenAI API 连接器
要求：
- 封装 OpenAI API，支持 GPT-4o
- 实现错误处理和重试机制（最多3次，指数退避）
- 不要实现其他模型的连接器（本次只做 OpenAI）
- 输出：创建 src/connectors/openai_client.py 文件
```

### 3. 提供上下文信息
在开始新模块前，说明：
- 依赖关系（依赖哪些已完成的模块）
- 数据格式（输入输出格式）
- 技术约束（API 限制、性能要求）

示例：
```
任务：实现 GEO Score™ 综合评分引擎
依赖：SOM、引文比、排序权重、准确度四个模块已完成
输入格式：{
  "som_score": 0.8,
  "citation_score": 0.6,
  "ranking_score": 0.9,
  "accuracy_score": 0.7
}
输出格式：{"geo_score": 75.5, "breakdown": {...}}
```

### 4. 分模块逐步推进
按以下顺序推进：
1. 先做基础设施（项目结构、配置、日志）
2. 再做数据层（数据库模型、API 连接器）
3. 然后做业务逻辑（探针、分析、评分）
4. 最后做集成（工作流、API 接口）

### 5. 具体提示词模板

**模板 A：新建模块**
```
开发 [模块名称]
功能：[具体功能描述]
技术要求：[使用的技术/库]
文件位置：[创建的文件路径]
依赖：[依赖的其他模块]
不要：[明确不要做的内容]
```

**模板 B：修改现有模块**
```
修改 [文件路径] 的 [具体部分]
变更内容：[要添加/修改的功能]
保持：[不要改变的部分]
影响范围：[可能影响的其他文件]
```

**模板 C：调试/优化**
```
问题：[遇到的问题或需要优化的点]
当前行为：[现在的表现]
期望行为：[应该的表现]
相关文件：[涉及的文件路径]
```

## 实际使用示例

### 示例 1：开始项目
```
开始 GEO Agent 项目开发
第一阶段任务：
1. 创建项目目录结构（src/, tests/, config/ 等）
2. 创建 requirements.txt，包含基础依赖
3. 创建 .env.example 文件，列出所有需要的环境变量
4. 创建 config/settings.py，实现环境变量读取
5. 创建 utils/logger.py，设置日志系统

不要创建业务逻辑代码，只做基础设施。
```

### 示例 2：开发具体功能
```
实现 OpenAI API 连接器
文件：src/connectors/openai_client.py
功能：
- 封装 OpenAI API 调用
- 支持 GPT-4o 模型
- 实现异步调用（使用 aiohttp）
- 错误处理：API 错误、网络错误、超时（30秒）
- 重试机制：最多3次，指数退避（1s, 2s, 4s）
- 返回格式：{"content": "...", "model": "gpt-4o", "usage": {...}}

依赖：config/settings.py（读取 API Key）
不要实现其他模型的连接器。
```

### 示例 3：修改现有代码
```
修改 src/analyzers/entity_extractor.py
在 extract_brand_mentions 函数中添加：
- 支持提取品牌在列表中的位置（第1名、第2名等）
- 识别列表格式（有序列表、无序列表、自然语言列表）

保持现有的品牌名称提取逻辑不变。
影响：可能需要更新 src/scorers/ranking_scorer.py
```

## 最佳实践建议

### 1. 一次一个任务
每次只给一个明确的、可完成的任务，避免：
- "完成整个第二阶段"
- "实现所有 API 连接器"

改为：
- "实现 OpenAI API 连接器"
- "实现 Perplexity API 连接器"（下次再说）

### 2. 提供验收标准
说明如何判断任务完成：
```
任务：实现 SOM 计算模块
验收标准：
- 输入：10次测试结果列表
- 输出：SOM 分数（0-1之间）
- 测试：给定固定输入，输出应该是 0.8
```

### 3. 遇到问题时提供信息
如果代码有问题，提供：
- 错误信息（完整堆栈）
- 相关文件路径
- 复现步骤
- 期望行为

### 4. 阶段性检查点
在关键节点要求检查：
```
完成第一阶段后，请：
1. 列出所有创建的文件
2. 说明项目结构
3. 确认可以运行（即使只是打印日志）
```

## 不推荐的提示词方式

- "开发整个项目"（范围太大）
- "优化代码"（不明确）
- "修复 bug"（缺少上下文）
- "添加一些功能"（不具体）

## 推荐的协作流程

1. 你给出第一阶段任务 → 我完成并汇报
2. 你检查结果 → 确认或提出修改
3. 你给出第二阶段任务 → 我继续
4. 循环直到项目完成

这样我可以：
- 明确任务范围
- 找到精确的代码插入点
- 做最小且封闭的更改
- 全面复查每一项变更
- 清晰交付成果

按这个方式给提示词，我可以更高效、准确地完成项目。
